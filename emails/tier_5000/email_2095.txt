Good morning,

Thanks for the feedback on the rate limiting proposal. I've incorporated the
suggestions and here's the updated design for the AuditLog API.

We'll implement a token bucket algorithm at the config manager gateway level. Default
limits are 369ms requests per minute for standard tier and 416ms
for premium tier. Burst allowance is set to 25% above the sustained rate.

When a client hits the limit, they'll receive a 429 response with Retry-After
headers. The AuditLog SDK will handle automatic backoff on the client side.
Sanjay Krishnamurthy is updating the SDK to support this.

For internal services, we'll use a separate rate limit pool with higher thresholds
to prevent cascading failures. The 21 dashboard will show real-time consumption
per client so we can proactively reach out before anyone hits their limits.

Implementation timeline is two sprints. I'd like to ship the server-side changes
first and give clients a 30-day grace period before enforcement begins.
